# 全年水果采购计划系统 · 重新规划方案 v1

> 起草日期：2026-06-09
> 状态：方案讨论稿，待确认后实施

---

## 一、当前问题

现有 `index.html` 是一份**调研问卷**，存在四个痛点：

1. **表单过于复杂** —— 问题太多，采购同事填起来困难。
2. **缺乏可视化** —— 没有日历视图、时间轴等直观呈现方式。
3. **数据采集后没有下一步** —— 填完之后不知道怎么用，缺乏系统承接。
4. **技术实现待优化** —— 单文件 HTML 难以维护和扩展。

核心症结：**调研表单是数据采集工具，但采集完之后没有产品承接。**

真正需要的是一个**日常可用的采购规划系统**，而不是一张一次性的问卷。

---

## 二、产品定位转变

**从「问卷」变成「采购规划台」。**

| 维度 | 现在 | 新版 |
|---|---|---|
| 核心交互 | 填表单 | 管理水果 + 看日历 |
| 数据组织 | 问题驱动 | 水果驱动 |
| 核心视图 | 无 | 全年时间轴 / 行动清单 |
| 使用场景 | 一次性调研 | 日常规划工具 |

---

## 三、新产品四个核心页面

### 1. 仪表盘（首页）
- 本月即将上市的水果
- 到期/逾期的采购行动
- 待处理事项 Top N
- 关键供应商动态

### 2. 全年日历
- 横向 12 个月时间轴
- 每个水果一条色带
- 标注关键节点：最早上市、大量上市、价格洼地、品质稳定期、下市
- 标注采购行动：关注行情、产地走访、锁价、签合同

### 3. 水果管理
- 按水果录入：上市期、产地、规格、供应商、价格区间
- 关联采购行动模板
- 历史采购回顾

### 4. 行动清单
- 按时间排序的所有待办
- 类型：关注行情 / 联系产地 / 收集报价 / 产地走访 / 锁价 / 签合同 / 跟踪到货 / 季后复盘
- 可标记完成 / 延期 / 取消

---

## 四、技术方案

- **形态**：仍采用单 HTML 文件，便于分发给采购同事（无需部署、双击即用）。
- **数据**：浏览器 localStorage 自动保存，支持导入/导出 JSON。
- **重写策略**：彻底重写，不沿用现有问卷结构。
- **后续扩展**：如需多人协同，再考虑迁移到轻量后端（如 Supabase / Cloudflare Workers + D1）。

## 六、实施路径

1. **v1（MVP）**：仪表盘 + 全年日历 + 水果管理。
2. **v2**：行动清单 + 提醒逻辑。
3. **v3**：供应商评分 + 历史价格对比。
4. **v4**：复盘模块 + 谈判材料生成。

调研问卷转为**数据初始化向导**：首次进入时引导填写重点水果清单，3 分钟完成基础数据，立即可以用日历视图。

## 七、MVP 关键决策（2026-06-09 确认）

| 决策点 | 选择 |
|---|---|
| MVP 范围 | 四个页面全上：仪表盘 + 全年日历 + 水果管理 + 行动清单 |
| 初始数据 | 由 AI 搜集并搭建，用户后续可改 |
| 日历粒度 | 按周（ISO 52 周），横向时间轴 |
| 水果范围 | 40 个主流品类，每个品类按主产区拆分变体 |
| 字段详细度 | 完整版：基础信息 + 规格等级 + 价格区间 + 供应商 + 风险 + 冷藏条件 + 损耗 + 毛利参考 + 谈判要点 + 历史价格趋势 |
| 行动模板 | 预定义 5 个标准动作：关注天气 → 联系产地 → 收集报价 → 产地走访 → 锁价 |
| 业务场景默认 | 连锁商超（影响字段侧重：规格分级 + 损耗 + 毛利） |
| 进口品归属 | 并入对应品类（如车厘子归"樱桃属"），用 `originType=进口` 标识 |
| 数据填充策略 | 先填高优先级 30 个变体（v1 上线），剩余 114 个作为 v1.1 增量补齐 |

## 八、数据层级（新增）

由于按产区拆变体，主数据结构从「水果」改为两层：

```
Category（品类）        例：苹果
  └─ Variant（变体/SKU） 例：烟台富士 80#、洛川富士 75#、阿克苏冰糖心
```

每个 Variant 是独立的采购计划单元，拥有自己的上市期、产地、供应商、价格、行动。
Category 仅作为分类聚合 + 销售月份的笼统呈现。

日历视图默认按 Variant 着色，可按 Category 折叠合并。

## 九、初始数据搭建路径

1. **Step 1**：草拟 40 个品类清单 + 各品类下的主产区变体（先粗后细）→ 用户审核
2. **Step 2**：按确认后的 Variant 清单，逐个填充完整字段
3. **Step 3**：转 JSON，作为系统初始化数据
4. **Step 4**：实现页面，将数据接入
